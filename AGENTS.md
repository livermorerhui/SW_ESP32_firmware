# SW ESP32 Firmware Rules

当前目录是 SonicWave ESP32 固件工作区。

## 工程目标

- 保持固件行为稳定、可测、可回滚
- 优先做状态机、模块边界、协议边界清晰的实现
- 禁止为了临时跑通而把日志、协议、校准、判断逻辑揉成一团

## 技术与结构

- 当前工程基于 `PlatformIO`
- 目标板和框架以 `platformio.ini` 为准
- 修改前先识别影响范围：
  - `src/` 主业务逻辑
  - `lib/` 公共模块
  - `include/` 头文件和接口
  - `data/` 资源或调试数据

## 固件开发规则

- 协议字段、命令字、状态码变更必须先明确兼容影响
- 设备状态判断、节律判断、告警逻辑优先模块化，不要散落在多个任务循环里
- 日志要服务于定位问题，避免无节制刷屏
- 时序、阈值、去抖、超时优先参数化，避免魔法数字到处散落
- 校准、存储、BLE 通信、业务判断尽量分层，降低耦合

## 与 SW 主项目联动

- 当前固件通常与 `SW` 仓库中的 `android / backend-api / admin-web` 联动开发；凡是影响 BLE、设备状态、控制指令、告警状态、配置同步的改动，都按跨仓联调任务处理
- 固件改动前必须先确认：
  - 哪个命令或字段变了
  - 类型、单位、取值范围、默认值是否变化
  - 是否会影响连接生命周期、状态机或错误码语义
- 禁止只在固件侧偷偷调整协议含义；任何跨端字段、命令字、状态枚举变化，都必须显式说明需要哪一端同步
- 变更协议或状态语义时，必须说明：
  - 是否兼容当前 APP / 后端版本
  - 是否需要兼容旧客户端
  - 推荐联调版本组合
- 联调验证至少覆盖：
  - 扫描、连接、重连、断开、超时
  - 核心指令收发
  - 关键状态上报
  - 异常路径和降级行为
- 日志必须服务于跨端排障，关键事件至少能对齐到：
  - 收到指令
  - 执行结果
  - 状态变化
  - 错误原因
- 临时兼容逻辑、协议兜底、旧版适配必须明确标注，不允许以隐藏分支长期存在

## 当前协作新增长期规则

- 对外状态语义必须区分：
  - 正式合同
  - warning/debug/兼容值
- 任何 reason / state / effect 如果只是 `WARNING_ONLY`，必须明确标注；下游仓不得自行升级成正式主分类
- 如果固件未来需要对外提供“稳定分类 + 具体实例名称”，必须显式区分：
  - 可长期依赖的稳定枚举/ID
  - 可变化的显示名/实例名
  - 禁止把会变化的显示名伪装成稳定协议分类
- 协议、状态枚举、原因码变化时，必须说明：
  - 哪些字段是下游可以正式依赖的
  - 哪些只是调试或兼容信息
- 如果固件行为变化会影响 APP / API / Web 的状态展示，必须明确：
  - 影响哪条链路
  - 需要同步哪些仓
- 关键日志必须能支撑跨仓排障，至少对齐：
  - 收到命令
  - 执行结果
  - 状态变化
  - 原因码 / 来源码
- 如果当前正式交付模型仍是 `APP_DRIVEN_SESSION`，则固件重构默认先冻结 BLE 外部合同，不得顺手改变：
  - execution model 语义
  - disconnect / reconnect 语义
  - `CAP? / SNAPSHOT / WAVE:* / EVT:*` 的正式字段、字段名和消费含义
  - Demo APP / SW APP 已正式依赖的兼容桥接输出
- 涉及 BLE 重构时，必须先对照 `docs/system/esp32_ble_safe_refactor_freeze_checklist.md` 审计，再决定哪些只允许做内部抽取、哪些属于正式协议变更
- 如果本轮开发尚未正式收官，除非用户明确提出，否则 AI 不得自行执行：
  - `git commit`
  - `git push`
  - 任何等价的远端发布动作
- 默认先完成：
  - 代码
  - 验证
  - 文档/报告
  再由用户决定是否提交、如何分组提交、是否推送
- 只要开发进入“需要真机测试”的阶段，AI 必须明确提出已经进入真机验证阶段，并先准备好自己后续分析需要的自动采集脚本或命令
- 真机测试阶段的默认分工固定为：
  - AI 负责准备日志采集、标记、停止、回收证据的脚本或命令
  - 用户负责真实设备操作，并从交互体感判断是否符合预期
  - 用户反馈“已测完 / 通过 / 不通过”后，AI 必须继续复核脚本采集到的固件 / APP / 服务器日志，不能只按体感结论直接收口
- 给用户真机测试步骤时，AI 必须先写清前置条件，至少包括：
  - 是否需要重装 APP
  - 是否需要重新烧录 ESP32 固件
  - 是否需要重启手机或 ESP32
  - 是否需要清 APP 数据、重新登录、连接服务器或切换网络
  - 如果以上都不需要，也必须明确写“不需要”，禁止让用户自行猜测
- 固件相关真机测试收口前，AI 至少要检查：
  - 本轮 capture 元数据和操作标记是否完整
  - APP / BLE 主链日志是否支持体感结论
  - ESP32 串口或设备侧日志是否可读、可用、能支撑结论
  - 如本轮约定采服务器日志，还要检查服务器日志是否真实有内容
- 如果某一路日志只有文件、没有可读内容，或内容不足以支撑结论，AI 必须明确标记该路证据未通过，不能笼统写成“测试通过”
- 修改串口采集、PlatformIO monitor、烧录、真机日志、BLE 回放等工具链前，AI 必须先搜索并审计成熟方案，优先使用 PlatformIO / 官方工具 / 项目既有脚本；禁止未经对比就自写底层串口或采集器并让用户反复真机试错
- 如确需自写工具，必须先说明官方方案为何不适用、替代哪一层能力、如何验证、如何回退

## 验证

- 优先运行 `pio` 相关构建 / 测试 / monitor 验证
- 如无法本地验证硬件行为，要说明需要哪块板、哪条链路、看哪些日志
