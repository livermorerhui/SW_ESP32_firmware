# Demo APP 校准模型比较说明

## 文档定位

本文专门说明 Task-C.1 之后 Demo APP 中的校准散点图与线性 / 二次模型比较区。

目标不是自动替你选模型，而是让工程判断过程更直观。

## 图上分别代表什么

- 蓝色散点：已经成功记录的校准点
- 线性曲线：基于当前点集计算的一次拟合结果
- 二次曲线：基于当前点集计算的二次拟合结果

当前图中采用：

- X 轴：记录回来的 `距离(mm)`
- Y 轴：外部已知 `参考重量(kg)`

内部拟合时，APP 仍会把 `distance_mm / 100.0` 转成与固件一致的运行时距离单位，确保 `ref/c0/c1/c2` 的解释与设备侧一致。

## 为什么这样画

这样做是为了同时满足两件事：

1. 图上阅读尽量直观
2. 拟合出来的工程系数仍与固件部署语义一致

也就是说：

- 图给人看
- 单位换算给模型对齐用

## 线性和二次应该怎么比较

优先看视觉贴合程度：

- 散点是否大致沿一条线分布
- 二次曲线是否真正更贴近中间和两端点位
- 曲线变化是否可解释，而不是只是更复杂

再看数值摘要：

- 样本点数
- 平均绝对误差
- 最大绝对误差
- 单调性检查

## 什么时候不要急着判断模型

如果出现下面情况，先不要急着定线性还是二次：

- 记录点太少
- 点位集中在很窄的重量区间
- 稳定体重还没真正收敛
- 散点出现明显离群或抖动样本

当前 Demo APP 会在点数不足时直接提示样本不足，避免“点太少但还在看曲线”的误导。

## 当前比较区的边界

Task-C.1 明确保持以下边界：

- 曲线比较 = 工程判断
- 记录校准点 = 收集样本
- 写入模型 / 校准 = 最终提交

因此，切换“线性 / 二次”只会改变当前比较重点和目标模型类型，不会自动写入设备。

## 拟合所有权说明

本仓当前的线性 / 二次拟合都在 Demo APP 本地完成，不在固件内完成。

也就是说：

- 散点图曲线来自 APP 本地对 `calibrationPoints` 的计算
- 固件负责的是当前部署模型的存取与运行时使用
- `获取模型` 读到的是设备当前模型
- 图上的线性 / 二次曲线不是设备返回的“候选拟合结果”

所以如果散点图为空，优先检查的是“有没有成功录到点”，而不是先怀疑拟合器本身。

Task-C.2 之后，录点区会直接显示录点条件和最近一次结果，因此在进入模型比较前，更容易先排除：

- 录点请求是否成功
- 是否因为未稳定而被设备拒绝
- 是否因为参考输入或样本状态有问题而没有进入点集

## Task-C.4 之后模型比较和部署如何衔接

现在模型比较区不只是“看图判断”，它还直接驱动主流程里的参数准备。

当前行为是：

1. 先看散点和两条拟合曲线
2. 再点击 `线性` 或 `二次`
3. Demo APP 自动把该模型的拟合结果准备成待写入参数
4. 最后点击 `写入模型`

所以现在的比较区与部署区关系更明确：

- 比较区负责帮助判断哪个模型更合理
- 选择区负责把那个判断转成待部署参数
- 写入模型负责最终下发

这也意味着手工填写 `ref/c0/c1/c2` 不再是主流程预期动作，而只是高级覆盖能力。

## Task-C.4b 之后比较区和部署区的过渡怎么读

Task-C.4b 没有改变比较本身，但把主流程的过渡顺序整理得更清楚了：

1. 先在录制阶段记录点
2. 看散点图和两条拟合曲线
3. 完成采样后点击 `停止录制`
4. 再选择 `线性 / 二次`
5. 最后点击 `写入模型`

这里的关键点是：

- `停止录制` 现在更明确地充当“结束采样、准备部署”的过渡动作
- 这不是因为模型写入在技术上必须先停录
- 而是为了让主流程读起来更自然，不把采样和部署混在一起

同时，`写入模型` 的完成也不再只靠 generic `ACK:OK` 心理推断，而是依赖显式结果：

- 成功：`ACK:CAL_SET_MODEL ...`
- 失败：`NACK:CAL_SET_MODEL ...`

这让“我刚刚选中的模型到底有没有真正部署成功”在主流程里更容易判断。
