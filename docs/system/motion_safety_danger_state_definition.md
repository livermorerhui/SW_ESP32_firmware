# Motion Safety 危险状态定义

## 文档目的

本文档用于正式固化 SonicWave 当前阶段对“危险状态”的定义，以及当前阶段真正关注的检测目标边界。

本文件是研究阶段的概念与范围定义，不是 firmware/runtime 规则说明，也不是最终产品承诺。

## 产品安全目标

当前阶段的产品安全目标不是做泛化的“跌倒识别”，而是识别这样一类高风险状态：

- 用户在设备运行过程中出现明显不适、失去正常姿态、发生危险倒伏
- 身体仍然部分留在平台或与平台支撑区域存在结构性耦合
- 设备持续振动可能造成重复机械接触、持续受力或姿态恶化
- 从而增加二次伤害风险

对老年用户、体弱用户、平衡能力较差用户，这一风险表述尤其重要。

## 正式危险状态定义

### 危险状态

当前项目语境中的“危险状态”，指的是：

> 用户在设备运行期间进入异常且不安全的身体姿态或倒伏状态，虽然尚未完全离开平台，但已经失去正常使用姿态，并且持续振动可能进一步造成二次伤害。

### 危险倒伏状态

“危险倒伏状态”是危险状态中的核心子集，强调：

- 姿态已经发生明显异常塌陷或支撑失效
- 不是正常律动中的短时波动
- 不是可立即自行恢复的轻微姿态调整
- 仍处于“未完全离台但已危险”的区间

### 未离台危险状态

当前阶段重点关注的就是“未离台危险状态”：

- 用户尚未完全脱离平台
- 或虽已部分卸载，但身体仍与平台/平台相关支撑区耦合
- 危险性来自“持续振动作用在异常姿态上”

## 当前主检测目标

当前主检测目标不是“所有跌倒”，而是：

- 危险异常倒伏
- 危险塌陷
- 危险失姿
- 且身体仍部分留台或仍与平台支撑结构相关联

这一目标可以概括为：

> 识别运行中仍未完全离台的危险倒伏状态，并尽量与正常使用样本区分开。

## 非目标 / 非当前阶段非目标

当前阶段明确不是以下目标：

- 泛化的全场景跌倒识别
- 医学意义上的摔倒诊断
- 完整姿态识别系统
- 通用人体动作识别
- 所有离台行为与所有危险行为的精细统一分类

尤其不是：

- “只要像跌倒就全部归为同一类”
- “只要承重很低就一定是危险状态”

## 与离台处理的边界

当前需要明确区分两个概念：

### 离台处理

离台处理更接近：

- 用户明显离开平台
- 设备可通过 leave-platform 路径处理

### 危险状态目标

危险状态目标更接近：

- 用户尚未完全离台
- 但姿态已经危险
- 平台持续振动会继续作用于异常姿态

因此，当前阶段的重点不是替代 leave-platform 行为，而是补足 leave-platform 尚不能覆盖的“未离台危险状态”。

## 为什么持续振动是问题核心

当前问题不是静态姿态分类，而是运行中的安全风险分类。

之所以要强调持续振动，是因为：

- 振动会持续施加机械刺激
- 身体已经失去正常姿态后，局部受力和重复接触会加重风险
- 用户可能因无力恢复姿态而继续暴露于风险中

因此，当前研究目标更准确地说是：

> 识别“振动持续作用下的未离台危险倒伏状态”。

## 为什么当前问题不是 generic fall detection

当前项目不应把自己描述为 generic fall detection，原因包括：

1. 目标空间更窄
   - 关注设备运行期间的特定危险状态
2. 风险来源不同
   - 关键风险是持续振动导致二次伤害
3. 边界不同
   - fully off-platform 情况并不是当前阶段主要优化对象
4. 结构不同
   - `FALL_ON_PLATFORM` 并非单一模式，而是存在 A/B 结构差异

## 当前阶段结论

当前阶段应正式采用如下表述：

- 当前主问题不是“泛化跌倒检测”
- 当前主问题是“未离台危险状态检测”
- 当前主目标是识别运行中仍与平台支撑关系相关联的危险倒伏状态
- leave-platform 边界仍然重要，但尚未完成与危险状态的细粒度分离

## 使用边界

本文档用于：

- 研究阶段统一口径
- 下一阶段实现评审的边界参考
- 避免把 v0.3 误写成正式 runtime 规则

- 宣布 production readiness
