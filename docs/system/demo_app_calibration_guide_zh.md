# Demo APP 校准工具使用说明

## 文档定位

本说明面向工程调试与模型比较，不是消费级向导。

校准工具的目标是：

- 观察当前距离/体重/稳定体重是否可信
- 记录样本
- 对比模型
- 支撑后续线性/二次模型分析

## 1. 校准工具的用途

校准工具主要用于三件事：

1. 看实时状态是否已经具备采样条件
2. 记录参考重量样本
3. 读取或写入当前模型参数

它不是“一键自动得出最终模型”的产品功能。

## 2. 当前阶段应该先看什么

在做任何模型拟合前，先看：

- 当前距离是否在持续刷新
- 当前体重是否随着加载变化
- 稳定体重是否能在波动收敛后出现
- 遥测曲线是否存在明显异常跳变或长期噪声

如果这些基础信号都不稳定，就不应急着判断模型优劣。

## 3. 如何观察距离 / 体重 / 稳定体重

### 当前距离

- 用来观察测量链路是否活着
- 对位移/姿态变化是否有响应

### 当前体重

- 用来观察实时估计是否连续、是否有明显波动

### 稳定体重

- 是固件在窗口足够稳定后给出的结果
- 更适合作为样本质量判断依据

建议：

- 先看实时体重是否趋于收敛
- 再看稳定体重是否出现
- 再决定是否记录校准点

## 4. 如何用遥测曲线辅助判断稳定性

遥测曲线的价值不在“好看”，而在快速回答几个工程问题：

- 测量流是否还在持续刷新
- 距离和体重是否同步变化
- 波动是否逐渐收敛
- 是否存在明显噪声、断流或异常跳变

经验上：

- 曲线持续刷新，说明流还活着
- 波动逐渐变小，说明更接近稳定
- 大幅跳变或频繁抖动，说明此时不适合记录样本

## 5. 为什么当前阶段不要急于确定最终模型

当前阶段首先要确认的是：

- 语义正确
- 数据可信
- 样本稳定

如果测量链路、稳定判定或样本质量还没站稳，过早固定最终模型容易把错误语义或脏样本写进模型。

所以顺序应当是：

1. 先确认链路正常
2. 再确认样本稳定
3. 再比较模型

## 6. 线性模型 / 二次模型比较的工程思路

当前 Demo APP 里的模型工具更适合做工程比较，而不是自动裁决。

大致思路：

- 线性模型
  - 参数少
  - 易解释
  - 适合先建立基线
- 二次模型
  - 拟合能力更强
  - 需要更小心地确认样本质量与单调性

建议流程：

1. 先建立可工作的线性基线
2. 记录多组可信样本
3. 再比较二次模型是否真的带来稳定且可解释的改进

## 7. 当前仍然保留的工程属性

Task-C 不会把校准工具改成消费级向导，因此仍然保留：

- 模型类型
- 参考值
- 系数 `c0/c1/c2`
- 最近校准点原始摘要
- 录制文件信息

## 8. Task-C.1 之后的推荐操作顺序

当前 Demo APP 校准区建议按下面的顺序使用：

1. 先在空载状态下执行“归零”
2. 点击“开始录制”
3. 放置已知重量并等待实时值 / 稳定体重收敛
4. 先看 Step 3 里的录点条件与录点摘要，再确认参考重量和实时距离，最后点击“记录校准点”
5. 重复多组点位后点击“停止录制”
6. 查看散点图与线性 / 二次拟合曲线
7. 最后再决定是否执行“校准”或“写入模型”

这里的关键边界是：

- “记录校准点”只是把当前稳定样本纳入比较数据集
- “校准 / 写入模型”才是最终应用到设备的步骤

## 9. 现在如何确认样本真的记录成功

Task-C.1 之后，记录成功不再只依赖瞬时提示，而会同时出现在界面上的持久信息里：

- 已记录校准点数量
- 最近一次记录点摘要
- 校准点列表
- 散点图中的新增点

因此，记录动作是否生效不需要再靠猜测。

## 9.1 现在 Step 3 录点区怎么读

Task-C.4e 已回滚 Task-C.4d 加进去的“录点前上下文”小区块。

现在 Step 3 会按下面这个顺序组织成一个连续的录点工作区：

1. 录点条件
2. 录点摘要
3. 参考重量输入框
4. 实时距离
5. 记录校准点按钮

这样用户可以更自然地按顺序完成录点：

- 先看当前能不能录
- 再看最近一次录点摘要
- 然后输入或修正参考重量
- 再确认当前实时距离
- 最后再按“记录校准点”

顶部原有的全局实时距离 / 体重 / 稳定体重显示仍然保留；这次调整只是把 Step 3 内部顺序改回正确方向，没有改动录点逻辑。

## 10. 工程工具为什么默认直接可见

当前阶段优先解决的是可发现性问题，而不是界面极简。

因此工程区默认直接显示：

- 获取模型
- CAL:ZERO
- 模型参考值
- `c0 / c1 / c2`

这样做的目的，是让新维护者和现场调试人员能够立即看到：

- 设备当前模型是什么
- 手工输入和写入入口在哪里
- 当前比较结果和写回参数之间的边界在哪里

## 11. 如何结合散点图判断线性还是二次更合适

看图时建议先看三件事：

1. 散点是否分布稳定，是否存在明显离群点
2. 线性曲线是否已经足够贴近主要点群
3. 二次曲线是否带来清晰、可解释的改善，而不是只是更“弯”

图形判断是主手段，数值摘要是辅助：

- 平均绝对误差
- 最大绝对误差
- 单调性检查

如果二次模型没有明显改善，工程上通常仍优先保留更容易解释的线性基线。

这些内容对普通用户并不友好，但对工程联调、日志对照和模型回溯很重要。

## 12. 录点成功到底由谁确认

当前实现里，`记录校准点` 不是本地“先加一条再说”。

真实规则是：

1. APP 先发送 `CAL:CAPTURE`
2. 固件判断当前样本能不能收
3. 只有收到 `ACK:CAL_POINT`，界面上的点数、表格、散点图才会更新

因此如果点击后还是 `0`，首先要看的是：

- 当前是否真的稳定
- 当前测量是否有效
- 设备是否回了 `NACK`

而不是先假设表格或图表一定坏了。

## 13. 如果点击后没有新增点，现在应该看哪里

本次审计之后，录点区会直接显示持久状态信息。

建议按下面顺序判断：

1. 看录点按钮是否处于可用状态
2. 看录点区的状态提示是“已发送等待结果”还是明确失败原因
3. 看系统日志中的 `[CAL_AUDIT]` 行
4. 再看顶部 ACK/NACK 信息

当前常见失败语义包括：

- `NOT_STABLE`
- `INVALID_SAMPLE`
- `INVALID_STABLE_BASELINE`
- `INVALID_REFERENCE_WEIGHT`

这些情况下，点数、表格、散点图都不会增长，因为固件没有确认本次录点成功。

## 14. Demo APP 和 ESP32 在校准里各自负责什么

当前建议把职责这样理解：

### Demo APP 负责

- 校准交互流程
- 开始 / 停止录制
- 已记录点列表
- 散点图与拟合曲线
- 线性 / 二次比较
- 模型选择
- 最终决定何时写回模型

### ESP32 负责

- 采集实时测量
- 生成稳定体重
- 判断当前样本是否足够稳定、有效
- 存储已部署模型
- 使用已部署模型做运行时重量计算

因此 ESP32 当前不应该被理解成“校准分析主界面”或“拟合比较主控”，它更接近：

- 运行时测量与模型执行端
- 录点时的轻量样本验证端

## 15. Task-C.2 之后录点前能看到什么

录点区现在会在按钮旁边直接显示一组紧凑条件，帮助判断当前为什么能录或为什么大概率会失败。

当前可见条件包括：

- 已开始录制
- 设备已连接
- 当前已稳定
- 参考重量有效
- 实时测量可用

这些条件是给工程人员的快速判断，不是消费级向导步骤。

同时仍要注意：

- 设备会做最终稳定样本校验
- 所以即使前置条件大多满足，设备仍可能因为样本无效或稳定基线问题拒绝本次录点

## 16. Task-C.2 之后录点后能看到什么

点击“记录校准点”后，录点区会直接显示最近一次结果，而不需要只盯着日志或顶部 ACK。

可能看到的结果包括：

- 已发送请求，等待设备校验
- 录点成功：已记录第 N 个点
- 录点失败：当前数据未稳定
- 录点失败：当前样本无效
- 录点失败：稳定基线无效
- 录点失败：参考重量无效

必要时还会保留设备原始原因码，方便工程联调。

## 17. 关键命令现在应如何理解

### 记录校准点

- 含义：请求设备返回当前稳定样本
- 成功后：APP 才会把点追加到本地点表和散点图

### 获取模型

- 含义：读取设备当前已部署模型

### 写入模型

- 含义：把当前输入的 `ref/c0/c1/c2` 写入设备

### CAL:ZERO

- 含义：发送工程归零命令，刷新设备侧参考基准

### 校准

- 含义：沿用当前 `Z/K` 参数执行 `SCALE:CAL`
- 注意：它不是“把当前线性/二次比较结果自动提交”

## 18. 下一阶段推荐的录点方式

当前实现里，录点仍通过 firmware `CAL:CAPTURE` 成功回包后才真正追加到点集。

但从下一阶段设计看，更推荐的方向是：

1. 用户点击“记录校准点”
2. Demo APP 立即记录当前输入参考重量
3. Demo APP 立即记录当前界面上看到的实时距离快照
4. 可选地同时保存：
   - 当前实时体重
   - 当前稳定体重
   - 时间戳
   - 当前质量状态

这样做的原因是：

- 点集工作流本来就属于 APP
- 拟合与比较本来就属于 APP
- 不应继续把 firmware 稳定锁存作为 APP 录点的唯一入口

## 19. 稳定体重在下一阶段仍然有什么作用

稳定体重不会被删掉，它仍然应该保留给：

- 运行时质量判断
- 运行时显示
- 平台状态逻辑
- 安全逻辑
- 校准时的辅助参考

但设计上不应再把它理解成：

- 所有校准录点都必须等待的硬门禁

更准确的理解应当是：

- 稳定体重 = 高质量运行时参考
- 校准录点 = APP 采样工作流

二者应当协同，但不必再完全绑定。

## 20. Task-C.3 之后“记录校准点”到底做了什么

现在点击“记录校准点”时，Demo APP 会直接把当前界面上的本地快照记成一个校准点。

当前主录点路径至少会记录：

- 当前输入的参考重量
- 当前实时距离快照
- 当前时间戳

如果当时界面上也能看到对应信息，还会一并保留：

- 当前实时体重
- 当前稳定体重
- 当前样本质量可见性
- 当前录点来源标记

因此录点后的点数、点表、散点图、拟合输入都会立刻更新，而不需要再等待 firmware `ACK:CAL_POINT` 才出现新点。

## 21. Task-C.3 之后录点条件怎么理解

录点区现在把条件分成两类：

### 必需条件

- 设备已连接
- 已开始录制
- 参考重量有效
- 当前实时距离可用

这些条件决定“现在能不能把当前本地快照记成一个点”。

### 质量提示

- 稳定体重可见
- 当前样本质量看起来正常

这些条件只帮助工程判断当前样本是不是更值得记录，不再作为所有录点的统一硬门禁。

## 22. Stable weight 现在还保留什么作用

Task-C.3 没有删除 stable weight。

它仍然保留给：

- 运行时显示
- 运行时质量判断
- 平台状态 / 安全逻辑
- 校准时的辅助参考
- 录点元数据

但它不再决定：

- 本地点集能不能增长

也就是说：

- stable weight 继续属于 runtime/helper signal
- calibration point capture 现在由 APP 主导

## 23. CAL:CAPTURE 现在是什么地位

`CAL:CAPTURE` 没有被粗暴删除，但已经从主工作流降级。

现在更准确的理解是：

- 它是旧设备稳定快照路径
- 或兼容 / 工程验证路径

而不是：

- 日常“记录校准点”的主入口

如果未来继续精简协议语义，应该沿着“保留兼容，但不再主导点集写入”的方向推进。

## 24. Task-C.4 / Task-C.4b 之后主流程怎么走

主界面现在优先展示的流程是：

1. 归零
2. 开始录制
3. 记录校准点
4. 看点表、散点图和拟合曲线
5. 停止录制
6. 选择模型
7. 写入模型

这意味着正常校准时，不再需要先理解一整组低层工程命令才能完成主要工作。

## 25. 选择模型后现在会发生什么

当你点击：

- 线性
- 二次

Demo APP 会自动把对应拟合结果准备成待写入模型参数，也就是：

- ref
- c0
- c1
- c2

因此正常路径里不再要求手工抄写或手工录入这些系数。

重要边界仍然保持：

- 选择模型 = 自动准备参数
- 写入模型 = 真正部署到设备

选择本身不会立刻写设备。

## 26. 为什么主界面不再直接摆出 Z/K 和校准

因为这条旧 `SCALE:CAL` 路径和新模型写入路径放在一起时，非常容易造成：

- 不知道哪个才是最终动作
- 写完模型后又去点旧校准
- 把两条不同语义的 apply path 混用

所以现在：

- `Z/K + 校准` 仍保留
- 但它已经被移动到“高级工程区”

主界面只把“写入模型”作为模型驱动校准流程的最终动作。

## 27. 高级工程区现在保留什么

高级工程区里仍保留了这些能力：

- 获取模型
- CAL:ZERO
- 手动修改 ref/c0/c1/c2
- 旧 Z/K 校准路径

它们仍然有工程价值，但不再参与主流程的心智模型。

也就是说：

- 主流程面向“记录点 -> 比较模型 -> 选择模型 -> 写入模型”
- 高级区面向“调试、兼容、手动覆盖”

## 28. Task-C.4a / Task-C.4b 之后如何确认“写入模型”是否成功

现在点击“写入模型”后，主流程区域会直接显示写入结果，不再需要只看日志或顶部 ACK。

当前会看到三种状态之一：

- 正在写入某个模型
- 写入成功：已部署线性 / 二次模型
- 写入失败：附带失败说明或设备返回码

这个反馈会出现在“写入模型”按钮附近，并且按钮本身也会用不同颜色提示最近一次结果。

Task-C.4b 进一步把这个反馈闭环变成了明确协议语义：

- APP 发送 `CAL:SET_MODEL`
- firmware 返回显式结果
  - `ACK:CAL_SET_MODEL ...`
  - 或 `NACK:CAL_SET_MODEL ...`
- APP 用这个显式结果关闭 pending 状态

因此现在“按钮一直橙色但不确定到底有没有写进去”这类情况，至少在协议设计上已经有了明确 completion signal。

## 29. 这次反馈改动没有改变什么

Task-C.4a 没有改变“写入模型”的真实语义。

它仍然是：

- 把当前选中并已准备好的模型写到设备

它没有新增：

- 第二条隐藏 apply path
- 自动提交
- 额外的模型选择逻辑

Task-C.4b 也没有改变这一点。

它改变的是：

- completion signal 变得显式
- pending/success/failure 的闭环更可靠
- “停止录制”在主界面上被放到了更自然的采样结束位置

## 30. Task-C.4c 之后为什么“写入成功”更容易看见

Task-C.4c 没有再改协议链路，而是直接修正“人是否真的看得见结果”。

现在主流程里的写入区会：

- 在按钮上方保留一个粘性的写入结果摘要
- 用更直接的标题显示：
  - 写入状态：正在写入
  - 写入状态：成功
  - 写入状态：失败
- 在 pending 阶段把按钮文案改成“正在写入模型…”

这意味着即使实时测量还在继续刷新，用户也更容易一眼看出：

- 刚才那次写入到底成功了没有
- 成功的是线性还是二次模型
- 失败时是什么原因

## 31. Task-C.4c 之后为什么 raw console 安静了很多

之前 raw console 会默认显示很多高频流日志，例如：

- `EVT:STREAM:...`
- `RX-RAW ...`

这些日志对于深度调试有用，但在正常开发和交互时会淹没真正重要的信息。

现在默认行为是：

- 高距离/体重流日志不再刷屏
- 重要日志仍然保留，例如：
  - `CAL:SET_MODEL` 的 TX / RX
  - capture 成功/失败
  - 安全 / 故障状态变化
  - 关键 ACK / NACK

如果需要深度排查，仍然可以在“高级工程区”里主动打开：

- 详细测量日志

所以当前策略变成了：

- 默认安静
- 需要时再显式打开高频流
