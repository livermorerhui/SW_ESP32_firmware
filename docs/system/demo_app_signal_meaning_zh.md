# Demo APP 信号含义说明

## 文档定位

本文面向：

- 项目负责人
- 测试人员
- 新维护者
- 未来新的 AI 分析窗口

当前 Demo APP 仍然是工程/调试控制台，不是面向终端用户的产品 APP。

## 系统状态区字段说明

### 状态

- 中文主语义：当前设备运行阶段
- 次级信息：保留原始状态枚举，例如 `IDLE`、`ARMED`、`RUNNING`、`FAULT_STOP`

建议理解：

- 空闲：设备未进入运行前就绪
- 就绪：已满足运行前条件，可理解为工程上的 `ARMED/READY`
- 运行中：波形工作中
- 故障停止：发生异常或联锁后停止

### 故障

- 中文主语义：当前故障/告警主题
- 次级信息：保留原始故障代码名，例如 `USER_LEFT_PLATFORM`
- 另有单独故障码显示数值，例如 `100`

### 安全原因

- 中文主语义：触发安全信号的直接原因
- 次级信息：保留原始 reason code

### 影响

- 中文主语义：该安全原因对运行造成的效果
- 次级信息：保留原始 effect code

### 运行态

- 中文主语义：安全事件对应的运行状态解释
- 次级信息：保留原始 runtime code

### 波形态

- 中文主语义：安全事件对应的波形状态
- 次级信息：保留原始 wave code

### 工程含义

- 对当前安全信号的简短工程解释
- 用于帮助快速判断：
  - 是仅提示
  - 需要人工恢复
  - 还是需要明确重启/排查

### 信号来源

- 中文主语义：这条安全信息来自哪里
- 次级信息：保留具体工程来源标识

当前常见来源：

- 协议安全事件（`EVT:SAFETY`）
- BLE 传输层（`BLE transport`）

### 安全码

- 安全相关数值代码
- 通常与 reason code 对应，例如：
  - `100`
  - `101`
  - `102`
  - `200`

## 关键安全原因中文含义

### `USER_LEFT_PLATFORM`

- 中文：用户离开平台
- 工程含义：平台上已没有有效在位载荷

### `FALL_SUSPECTED`

- 中文：疑似摔倒
- 工程含义：体重变化速率或运行条件触发了摔倒怀疑逻辑

### `BLE_DISCONNECTED`

- 中文：蓝牙连接中断
- 工程含义：通信链路断开，需要关注是否需要重新连接

### `MEASUREMENT_UNAVAILABLE`

- 中文：测量不可用
- 工程含义：测量链路当前不可用或不可信，应先排查采样链路

## 关键影响中文含义

### `WARNING_ONLY`

- 中文：仅警告
- 含义：提示异常，但不必然强制停掉全部运行链路

### `RECOVERABLE_PAUSE`

- 中文：可恢复暂停
- 含义：当前需要暂停，处理原因后可以再恢复

### `ABNORMAL_STOP`

- 中文：异常停止
- 含义：应按异常处理，通常需要明确确认后再重新启动

## 运行态 / 波形态含义

### 运行态

- `IDLE` / 空闲
- `ARMED` / 就绪
- `RUNNING` / 运行中
- `FAULT_STOP` / 故障停止
- `DISCONNECTED` / 连接已断开

### 波形态

- `RUNNING` / 运行中
- `STOPPED` / 已停止
- `UNKNOWN` / 未知

## 典型组合示例

### 用户离开平台 -> 可恢复暂停

- 安全原因：用户离开平台（`USER_LEFT_PLATFORM`）
- 影响：可恢复暂停（`RECOVERABLE_PAUSE`）
- 典型理解：当前先停，重新在位后再恢复

### 疑似摔倒 -> 异常停止

- 安全原因：疑似摔倒（`FALL_SUSPECTED`）
- 影响：异常停止（`ABNORMAL_STOP`）
- 典型理解：优先按异常情况处理，不应直接无脑恢复

### 测量不可用 -> 仅警告

- 安全原因：测量不可用（`MEASUREMENT_UNAVAILABLE`）
- 影响：仅警告（`WARNING_ONLY`）
- 典型理解：先排查测量链路、Notify、固件输出和传感器状态

### 蓝牙连接中断 -> 连接链路异常提示

- 安全原因：蓝牙连接中断（`BLE_DISCONNECTED`）
- 影响：通常表现为仅警告
- 典型理解：首先看作通信链路异常提示，必要时重新连接

## 为什么还保留原始代码

Task-C 不会移除原始枚举/代码，因为 Demo APP 仍然承担工程联调职责。

保留原始代码的价值：

- 便于和固件日志逐项对照
- 便于协议问题排查
- 便于未来维护者或 AI 窗口追踪真实语义来源

最终策略：

- 中文主语义先帮助理解
- 原始代码作为次级信息帮助调试

## 校准区新增工程语义补充

Task-C.1 之后，校准区里新增了一批更明确的工程提示，建议按下面理解：

### 开始录制 / 停止录制

- 作用：控制当前是否允许记录校准点
- 不是：最终校准提交命令
- 额外作用：同时控制本地 CSV 遥测录制会话

### 记录校准点

- 作用：发送 `CAL:CAPTURE`，把当前稳定样本纳入校准点数据集
- 前提：只有录制已开始时才允许点击
- 成功后会同步更新：
  - 点数
  - 点列表
  - 散点图

### 获取模型

- 作用：从设备读取当前已部署模型参数
- 典型用途：回读确认当前设备里实际生效的 `type/ref/c0/c1/c2`

### 写入模型

- 作用：将当前界面里的模型参数写回设备
- 边界：这是最终提交动作，不是单纯比较动作

### CAL:ZERO

- 作用：发送底层工程归零命令
- 典型理解：刷新模型参考基准，而不是记录校准点

### 线性 / 二次

- 作用：切换当前比较重点与目标模型类型
- 不会：自动把拟合结果写回设备

### 模型参考值 / c0 / c1 / c2

- `模型参考值`：运行时参考距离 `ref`
- `c0`：二次项系数
- `c1`：一次项系数
- `c2`：常数项系数

这些字段仍然偏工程化，保留它们是为了：

- 和固件协议直接对照
- 便于手工写入和回读验证
- 支持未来维护者或 AI 窗口直接追踪真实部署参数
