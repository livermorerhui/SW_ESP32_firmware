# Motion Safety 研究规则工作版 v0.3

## v0.3 是什么

v0.3 是 SonicWave 当前阶段接受的**研究规则工作版**。

它的作用是：

- 帮助离线复核当前样本
- 作为当前研究阶段的可用工作基线
- 支撑对危险状态样本结构的继续理解

它不是：

- 正式 firmware 规则
- 正式 runtime 规则
- 生产可用 detector

## v0.3 的背景

在当前研究过程中，规则经历了从 v0 到 v0.2，再到 v0.3 的演进。

演进的核心原因不是“追求更复杂”，而是研究目标逐步收敛后，必须让规则更贴近当前真正关心的危险状态：

- 未离台危险状态
- A/B 两类危险倒伏

尤其是：

- B型：外部支撑分担型危险倒伏

推动了 v0.3 的必要性。

## 当前依据来源

当前对 v0.3 的书面描述，主要依据现有离线研究脚本：

- `tools/detect_v0.py`

在该脚本中，v0.3 被明确标注为当前阶段的研究策略版本，并保留了两条危险路径：

- `persistent_low_weight_no_recovery`
- `large_migration_no_recovery_v03`

这也说明 v0.3 当前仍属于研究侧的离线规则工作版，而不是 device runtime 的正式行为定义。

## 相比早期版本，v0.3 的关键变化

从当前离线脚本 `tools/detect_v0.py` 可见，v0.3 保留了两条核心危险路径：

1. 深低承重且持续未恢复路径
2. 放宽后的“异常迁移未恢复”路径

其中第二条路径的调整目标非常明确：

- 用于补强 B 型危险状态

脚本中同时保留了从 v0.2 到 v0.3 的关键参数调整说明，其核心含义不是“简单放宽阈值”，而是让“异常迁移未恢复”路径对 B 型样本更敏感。

也就是说，v0.3 的重要改进不是简单降低门槛，而是承认：

- 危险状态不一定都长成深低承重

## 为什么 v0.3 是必要的

如果没有 v0.3，当前研究会面临一个关键问题：

- 规则更容易只覆盖 A 型
- 而错过 B 型：外部支撑分担型危险倒伏

新采集到的 `FALL_ON_PLATFORM` 样本说明：

- 有些危险状态仍然危险
- 但尾段承重不一定足够低
- 如果只依赖深低承重语义，会漏掉真实危险样本

因此 v0.3 的必要性在于：

> 它让当前研究规则能够同时保留“深低承重危险路径”和“异常迁移未恢复路径”，从而更接近当前阶段的真实目标空间。

## v0.3 当前能够做到什么

以当前阶段的研究口径来说，v0.3 当前可以做到：

- 作为当前 accepted working baseline 使用
- 保留对更深危险状态的识别路径
- 对 B 型危险状态提供更好的覆盖能力
- 明确把当前主判别聚焦在：
  - `NORMAL`
  - `FALL_DANGER`

它的价值主要在研究流程里，而不是在生产流程里。

## v0.3 当前不能声称什么

v0.3 当前不能声称：

- 已经完成 danger vs leave-platform 的精细分离
- 已经是正式 runtime-ready 规则
- 已经可以直接作为 firmware 行为标准
- 已经完成全面 fall detection
- 已经达到 production readiness

尤其不能写成：

- “v0.3 就是最终摔倒检测逻辑”
- “v0.3 已可直接替换当前 runtime 逻辑”

## 为什么 v0.3 仍然只是研究规则工作版

原因包括：

1. 当前研究目标本身就不是泛化跌倒检测
2. 当前规则仍面向离线样本复核
3. leave-platform 与 danger-state 的细边界尚未完成
4. 当前尚未完成 runtime 参数化与正式验证闭环
5. 当前尚未完成 production 级风险评审

因此更准确的表述应是：

> v0.3 是当前阶段接受的研究规则工作版，而不是正式 runtime 规则。

这一定义也意味着：

- 它可以指导当前阶段的研究复核与评审
- 但不能直接变成 firmware 阈值或自动停机/暂停逻辑

## 当前接受边界

当前项目可以正式接受以下边界：

- v0.3 可作为当前研究阶段的工作基线
- v0.3 可用于帮助理解当前样本结构
- v0.3 可用于支持下一阶段受控评审与进一步研究
- v0.3 不得直接被描述为正式 firmware/runtime 逻辑

## 与下一阶段的关系

下一阶段如果继续推进，应基于 v0.3 的书面边界来工作：

### 可以做的

- 继续用 v0.3 作为研究参考基线
- 用更多样本检验 danger vs leave-platform 的边界
- 在 Demo APP / 标注 / 导出流程中提升研究支持效率

### 暂时不应做的

- 直接推广为 runtime 规则
- 直接改 firmware 阈值
- 直接改设备自动停机/暂停决策

## 当前正式表述

建议项目内统一使用如下表述：

> v0.3 是 SonicWave 当前阶段接受的研究规则工作版。它已经体现出 A/B 型危险倒伏结构下的当前理解，尤其补强了 B型：外部支撑分担型危险倒伏，但它仍未解决 danger vs leave-platform 的细粒度分离，因此不能被描述为正式 runtime-ready 规则。
